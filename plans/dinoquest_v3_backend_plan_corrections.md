## [CRITICAL-1] Auth Strategy — HTTP-only Cookie, NOT Bearer Token

### What the original plan said (WRONG)
> Use `Authorization: Bearer <token>` header. Store JWT in Zustand `authStore` in memory / localStorage.

### What the frontend actually tells us
`src/store/index.ts` line 18: The store holds a `user` object but **no `token` field**. The absence of a token field in the store is unambiguous — the frontend was designed for cookie-based auth where the browser manages the credential automatically, not the application code.

### Decision
**Use HTTP-only cookies.** This matches both the original rebuild plan (`dinoquest_v3_rebuild_plan_updated.md` line 87) and the actual frontend store shape.

### How it works with the Cloudflare deployment
The FastAPI API is exposed via **Cloudflare Tunnel** at `https://api.yourdomain.com`. The TanStack Start frontend runs on **Cloudflare Workers** at `https://dinoquest.yourdomain.com`. These are different origins, so the cookie must be `SameSite=None; Secure`. Cloudflare Tunnel terminates HTTPS, so the `Secure` requirement is automatically satisfied in production.

For **local development**, use the TanStack Start / Vite dev proxy so that `/api/*` requests appear same-origin and `SameSite=Lax` works without HTTPS:

```typescript
// vite.config.ts (add to existing config)
export default defineConfig({
  // ...existing config...
  server: {
    proxy: {
      '/api': 'http://localhost:8122',
      '/ws': { target: 'ws://localhost:8122', ws: true },
    },
  },
});
```

With this proxy, local dev hits `http://localhost:3000/api/...` which is same-origin — no CORS, `SameSite=Lax` cookies work fine.

### Corrected login endpoint

```python
# routers/auth.py

@router.post("/login", response_model=UserOut)
async def login(body: LoginRequest, response: Response, db: AsyncSession = Depends(get_db)):
    user = (await db.execute(
        select(User).where(User.username == body.username)
    )).scalar_one_or_none()
    if not user or not pwd_ctx.verify(body.password, user.hashed_password):
        raise HTTPException(401, "Invalid credentials")

    token = create_access_token(user.id)

    is_prod = settings.environment == "production"
    response.set_cookie(
        key="access_token",
        value=token,
        httponly=True,
        samesite="none" if is_prod else "lax",
        secure=is_prod,
        max_age=86400 * settings.access_token_expire_days,
    )
    # Return user object only — no token in body
    return UserOut.model_validate(user)

@router.post("/register", response_model=UserOut, status_code=201)
async def register(body: RegisterRequest, response: Response, db: AsyncSession = Depends(get_db)):
    # ... create user ...
    # Set cookie same as login
    response.set_cookie(key="access_token", value=create_access_token(user.id), ...)
    return UserOut.model_validate(user)

@router.delete("/logout", status_code=204)
async def logout(response: Response):
    response.delete_cookie("access_token")
```

### Corrected `get_current_user` dependency

```python
# dependencies.py — reads from cookie, NOT Authorization header

from fastapi import Cookie

async def get_current_user(
    access_token: str | None = Cookie(default=None),
    db: AsyncSession = Depends(get_db),
) -> User:
    if not access_token:
        raise HTTPException(status_code=401, detail="Not authenticated")
    try:
        payload = jwt.decode(access_token, settings.secret_key, algorithms=["HS256"])
        user_id: int = int(payload["sub"])
    except (JWTError, KeyError, ValueError):
        raise HTTPException(status_code=401, detail="Invalid or expired session")

    user = await db.get(User, user_id)
    if not user:
        raise HTTPException(status_code=401, detail="User not found")
    return user
```

## [CRITICAL-2] User Model JSON — Backend Must Return camelCase

### What the original plan said (WRONG)
```python
class UserOut(BaseModel):
    global_role: str  # snake_case
```

### What the frontend actually tells us
`src/store/index.ts` line 6: The frontend User type uses `globalRole` (camelCase). The frontend was generated by Lovable using TypeScript conventions where all fields are camelCase.

### Decision
The **backend stays Python snake_case internally** but all JSON responses must be serialized as **camelCase**. Use Pydantic's built-in alias generator. Apply this globally so no schema is missed.

### Implementation

**In `database.py` or a new `base_schema.py`:**

```python
from pydantic import BaseModel, ConfigDict
from pydantic.alias_generators import to_camel

class APIModel(BaseModel):
    """Base class for all request/response schemas. Serializes to camelCase."""
    model_config = ConfigDict(
        alias_generator=to_camel,
        populate_by_name=True,   # allows both snake_case and camelCase on input
        from_attributes=True,
    )
```

**All schemas inherit from `APIModel` instead of `BaseModel`:**

```python
# schemas/auth.py
from backend.base_schema import APIModel

class UserOut(APIModel):
    id: int
    username: str
    email: str | None
    global_role: str          # serialized as "globalRole" in JSON
    created_at: datetime      # serialized as "createdAt" in JSON
```

**All list/pagination wrappers also use `APIModel`:**

```python
class PaginatedResponse(APIModel):
    items: list[Any]
    next_cursor: str | None   # → "nextCursor" in JSON
    total: int
```

### Request bodies — accept camelCase from frontend
Since `populate_by_name=True` is set, the frontend can send either `global_role` or `globalRole` and Pydantic will accept both. This gives flexibility during frontend development.

### Impact on every router
No router code changes needed — Pydantic handles serialization. The `response_model=UserOut` declaration is sufficient; FastAPI calls `.model_dump(by_alias=True)` internally when serializing.

### Updated Frontend Integration Contract (replaces original)

The store in `src/store/index.ts` expects:
```typescript
interface User {
  id: number;
  username: string;
  email: string | null;
  globalRole: string;           // "user" | "superadmin"
  createdAt: string;            // ISO 8601
}
```

On login/register, the API returns just the `User` object (no token wrapper):
```json
{ "id": 1, "username": "alice", "email": null, "globalRole": "user", "createdAt": "2026-05-18T10:00:00Z" }
```

The browser stores the cookie automatically. The frontend store sets `user` from the response body.

## [CRITICAL-3] Route Structure — Flat UI Routes, Family ID via Store

### What the original plan said (WRONG)
> The frontend's TanStack Router file `_authenticated/families/$familyId/tests/$testId.tsx` maps to `/families/1/tests/5` — Backend routes use `/api/families/{family_id}/tests/{test_id}`.

### What the frontend actually tells us
- `src/routes/tests.index.tsx` line 8: The tests page is at `/tests` (flat, no `$familyId` segment)
- `src/routes/quests.index.tsx` line 8: The quests page is at `/quests` (flat)

The frontend URL structure is:
```
/login
/register
/families            # Family lobby
/families/$familyId  # Family detail
/quests              # Quest list (for active family)
/tests               # Test list (for active family)
/pets
/leaderboard
/rewards
```

There is no `$familyId` in the quest/test/pet/leaderboard routes. The active family is selected via a **FamilySwitcher** component and stored in the Zustand store. When making API calls, the frontend reads `store.activeFamilyId` and includes it in the request.

### Decision
**Keep the backend URL structure exactly as designed** (`/api/families/{family_id}/quests`, etc.). The family-scoped URLs are correct API design. The frontend simply reads `activeFamilyId` from its store and injects it into the API call URL.

This is a frontend concern, not a backend concern. The backend plan's API routes do **not** change.

### What changes in the frontend (document this for the frontend developer)

Every TanStack Query hook that fetches family-scoped data must read `activeFamilyId` from the store:

```typescript
// src/hooks/useQuests.ts
import { useStore } from '../store';
import { useQuery } from '@tanstack/react-query';

export function useQuests() {
  const activeFamilyId = useStore(s => s.activeFamilyId);

  return useQuery({
    queryKey: ['quests', activeFamilyId],
    queryFn: () => fetch(`/api/families/${activeFamilyId}/quests`).then(r => r.json()),
    enabled: !!activeFamilyId,  // Don't fetch if no family selected
  });
}
```

The `/quests` page renders a loading/empty state when `activeFamilyId` is null (user hasn't joined or selected a family yet).

### What this means for the backend
No changes to the backend route structure. The family-scoped URL pattern is correct and stays:
```
/api/families/{family_id}/quests
/api/families/{family_id}/tests
/api/families/{family_id}/pets
/api/families/{family_id}/leaderboard
/api/families/{family_id}/rewards
```

## [HIGH] Auth Strategy Conflict in Original Documents Resolved

The original `dinoquest_v3_rebuild_plan_updated.md` (line 87) says HTTP-only cookies. The backend plan switched to Bearer. **The rebuild plan was right.** Bearer is eliminated. See [CRITICAL-1] above.

**Summary of the final single decision:**

| Context | Strategy |
|---|---|
| Production (Cloudflare Tunnel) | HTTP-only cookie, `SameSite=None`, `Secure=True` |
| Local development | HTTP-only cookie, `SameSite=Lax`, `Secure=False`, Vite proxy |
| WebSocket (all environments) | JWT token as `?token=` query param (browser WS API cannot send cookies) |

The WebSocket endpoint is the **only place** where a JWT token is used explicitly. On WS connect, the frontend requests a short-lived WS token:

```
GET /api/auth/ws-token   → { wsToken: "<signed JWT, 1hr expiry>" }
```

The frontend then connects:
```
ws://api.host/ws/families/{id}?token=<wsToken>
```

This WS token endpoint requires the session cookie, same as every other protected endpoint. The WS token has a short TTL (1 hour) and is only valid for WS connections.

## [MEDIUM-2] Family Deletion vs Auto-Promote — Logic Corrected

### What the original plan said (wrong framing)
> `DELETE /api/families/{family_id}` — soft-delete (owner only, **triggers auto-promote**)

### What the spec actually says
Auto-promote applies when the **creator leaves** (removes themselves from membership). Deleting the family is a separate, terminal action — no promotion happens because the family stops existing.

### Corrected logic table

| Action | Trigger | Result |
|---|---|---|
| Owner removes themselves via `DELETE /api/families/{id}/members/{owner_id}` | `auto_promote_or_delete()` | Eldest other parent becomes owner; if none → soft-delete family |
| Owner calls `DELETE /api/families/{id}` | Direct soft-delete | Family is soft-deleted. No promotion. All members lose access. |
| Non-owner parent removes themselves | No promotion check needed | Member row deleted, family unaffected |

### Corrected `DELETE /api/families/{family_id}` logic

```python
@router.delete("/{family_id}", status_code=204)
async def delete_family(
    family_id: int,
    current_user: User = Depends(get_current_user),
    db: AsyncSession = Depends(get_db),
):
    family = await db.get(Family, family_id)
    if not family or family.is_deleted:
        raise HTTPException(404, "Family not found")
    if family.owner_id != current_user.id:
        raise HTTPException(403, "Only the family owner can delete the family")

    # Direct soft-delete — NO auto-promote
    await soft_delete_family(family_id, db)
    await write_audit(family_id, current_user.id, "family_deleted", {}, db)
    await db.commit()

    # Broadcast to family room so all connected clients redirect away
    await ws_manager.broadcast(family_id, "family_deleted", {"family_id": family_id})
```

### Corrected `DELETE /api/families/{family_id}/members/{user_id}` logic

```python
@router.delete("/{family_id}/members/{target_user_id}", status_code=204)
async def remove_member(
    family_id: int,
    target_user_id: int,
    parent: FamilyMember = Depends(require_parent),
    db: AsyncSession = Depends(get_db),
):
    # A parent removing themselves → check if they're the owner → auto-promote
    if target_user_id == parent.user_id:
        # Self-removal
        await _delete_member_row(family_id, target_user_id, db)
        family = await db.get(Family, family_id)
        if family.owner_id == target_user_id:
            await auto_promote_or_delete(family_id, target_user_id, db)  # ← HERE
    else:
        # Removing someone else
        target = await db.get(FamilyMember, (family_id, target_user_id))
        if not target:
            raise HTTPException(404, "Member not found")
        await _delete_member_row(family_id, target_user_id, db)
        await ws_manager.send_to_user(family_id, target_user_id, "kicked", {})

    await write_audit(family_id, parent.user_id, "member_removed", {"target_user_id": target_user_id}, db)
    await db.commit()
```

## [MEDIUM-3] Invite Code — Strictly 6 Numeric Digits

### What the original plan said (WRONG)
```python
def generate_invite_code() -> str:
    return secrets.token_urlsafe(4).upper()[:6]
    # Produces alphanumeric like "X7K2MQ" — NOT digits only
```

### What the spec says
`dinoquest_v3_rebuild_plan_updated.md` line 107: **"6-digit"** invite code — meaning numeric digits `000000`–`999999`.

### Corrected implementation

```python
import random

def generate_invite_code() -> str:
    """Returns a zero-padded 6-digit numeric string, e.g. '042819'."""
    return str(random.randint(0, 999999)).zfill(6)
```

### Schema change

The `code` column in `family_invites` must accommodate leading zeros:
```python
code: Mapped[str] = mapped_column(String(6), unique=True, nullable=False)
```

`String(6)` stores the string as-is. `"042819"` stays `"042819"`, not `42819`. This is already correct since it's a TEXT column in SQLite.

### Display in UI
The frontend renders the code as `042 819` (space in the middle for readability). The backend stores and validates it without the space — `042819`. The frontend strips spaces before sending the join request:

```typescript
// Frontend join request body
{ code: inviteCode.replace(/\s/g, '') }  // "042 819" → "042819"
```

## [MEDIUM-4] Deployment Model — Definitive Architecture

### The contradiction
- `wrangler.jsonc` + TanStack Start: Frontend is a **Cloudflare Workers SSR app**, not static HTML.
- Original backend plan: nginx serves `frontend_dist/` as static files.

These are incompatible. nginx cannot serve a Cloudflare Workers app.

### Definitive deployment architecture

```
┌─────────────────────────────────────────────────────┐
│  Internet                                            │
│                                                      │
│  User Browser                                        │
│       │                                              │
│       ├──── HTTPS ──► dinoquest.yourdomain.com       │
│       │                    │                         │
│       │              Cloudflare Workers              │
│       │              (TanStack Start SSR)            │
│       │              renders pages,                  │
│       │              forwards API calls              │
│       │                    │                         │
│       │              HTTPS (Cloudflare Tunnel)       │
│       │                    │                         │
│       └──── WS ────► api.yourdomain.com              │
│                            │                         │
│                      Cloudflare Tunnel               │
│                            │                         │
└─────────────────────────── │ ───────────────────────┘
                             │
              ┌──────────────▼──────────────┐
              │  Home Server (Tanix W2)      │
              │                             │
              │  Docker Compose             │
              │  ┌──────────────────────┐   │
              │  │  api (FastAPI)        │   │
              │  │  port 8122           │   │
              │  └──────────────────────┘   │
              │  ┌──────────────────────┐   │
              │  │  data/ (SQLite)      │   │
              │  └──────────────────────┘   │
              └─────────────────────────────┘
```

### What this means concretely

**The `nginx` service and `frontend` build container are removed from `docker-compose.yml`.**

The frontend is deployed separately to Cloudflare Workers via `wrangler deploy` or Cloudflare Pages CI. It is not part of the Docker Compose stack at all.

**New `docker-compose.yml` (backend only):**
```yaml
version: "3.9"
services:
  api:
    build:
      context: .
      dockerfile: Dockerfile.api
    env_file: .env
    ports:
      - "8122:8122"    # Cloudflare Tunnel connects to this port
    volumes:
      - ./data:/app/data
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "python", "-c", "import urllib.request; urllib.request.urlopen('http://localhost:8122/api/health')"]
      interval: 30s
      timeout: 10s
      retries: 3

volumes: {}
```

**Cloudflare Tunnel config (`~/.cloudflared/config.yml` on home server):**
```yaml

ingress:
  - hostname: api.yourdomain.com
    service: http://localhost:8122
  - service: http_status:404
```

**Frontend deployment:**
```bash

```

Frontend environment variable pointing at the API:

### Cookie configuration for this architecture

Since `dinoquest.yourdomain.com` and `api.yourdomain.com` are different origins, the cookie **must** be:

```python
response.set_cookie(
    key="access_token",
    value=token,
    httponly=True,
    samesite="none",    # Required for cross-site cookies
    secure=True,        # Required when SameSite=None
    max_age=86400 * settings.access_token_expire_days,
    domain=".yourdomain.com",  # Optional: allows sharing across subdomains
)
```

Add to `.env`:
```env
COOKIE_DOMAIN=.yourdomain.com    # Your actual domain; leave empty for localhost
```

And in `config.py`:
```python
cookie_domain: str = ""          # Empty string = no domain attribute (localhost dev)
```

### CORS configuration for this architecture

Since frontend and API are different origins, **CORS is now required in production too** (not just dev):

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=settings.allowed_origins,   # ["https://dinoquest.yourdomain.com", "http://localhost:3000"]
    allow_credentials=True,                   # Required for cookies
    allow_methods=["*"],
    allow_headers=["*"],
)
```

`.env` in production:
```env
ALLOWED_ORIGINS=https://dinoquest.yourdomain.com
```

`.env` in development:
```env
ALLOWED_ORIGINS=http://localhost:3000,http://localhost:5173
```
